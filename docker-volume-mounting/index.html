DOCKER VOLUMES

docker_mount_cwd_to_workspace — Docker Volume Mount Guide

Mount your current working directory into a Docker container's /workspace with one config option. This guide covers the exact syntax, common permission errors (UID/GID mismatch), and verified working examples for Hermes WebUI, Claude Code, and general Docker Compose setups.

What Is docker_mount_cwd_to_workspace?

docker_mount_cwd_to_workspace is a configuration flag used in AI coding tools and development containers. When enabled, Docker mounts your current working directory ($PWD) from the host into the container at /workspace. This gives the container read/write access to your local project files — critical for tools that need to read your codebase, edit files, or run commands in your project context.

In Hermes WebUI, this is handled by the volume mount configuration in the compose files. In Claude Code, it's controlled by the docker_mount_cwd_to_workspace setting.

How It Works: The Docker Volume Syntax

Under the hood, this flag generates a standard Docker bind mount. Here's what it's equivalent to:

docker_mount_cwd_to_workspace = true (what the tool does)

# Equivalent Docker run command: docker run -v "$(pwd):/workspace" container-name # Equivalent Docker Compose: services: my-service: volumes: - ./:/workspace # Mount cwd into /workspace

docker_mount_cwd_to_workspace = false (explicit path instead)

# Mount a specific directory, not cwd: volumes: - /home/user/projects/myapp:/workspace

Hermes WebUI: Volume Mount Configuration

Hermes WebUI's multi-container setups use a shared data directory pattern. The workspace mount is separate from the agent config mount:

# Single-container — cwd mounted as workspace services: hermes-webui: volumes: - ~/.hermes:/home/hermeswebui/.hermes # Agent config + sessions - ~/workspace:/workspace # Your files, mounted into container # Three-container — shared data dir replaces cwd volumes: - /mnt/data/hermes-allinone:/home/hermeswebui/.hermes - /mnt/data/hermes-allinone/hermes-agent-src:/home/hermeswebui/.hermes/hermes-agent
The #1 failure point: UID/GID mismatch. The container user (hermeswebui, UID 1000) needs write permission on the host directory. On macOS, id -u returns 501 — the default WANTED_UID=10000 won't match. Always set WANTED_UID and WANTED_GID in .env before starting.

Common docker_mount_cwd_to_workspace Errors

Error 1: Permission Denied on /workspace

ERROR: Permission denied: '/workspace' ERROR: Cannot write to mounted directory

Cause: Container user UID doesn't match host directory owner.

Fix: Run id -u on the host, set WANTED_UID=<your-uid> in .env, then docker compose down && docker compose up -d.

Error 2: Mounted Directory Is Empty in Container

$ ls /workspace # (nothing shown)

Cause: The host path doesn't exist, or Docker Desktop hasn't been granted file-sharing permissions for that path.

Fix: Docker Desktop → Settings → Resources → File Sharing → add your project directory. On Linux, verify the path exists: ls -la $PWD.

Error 3: Changes in Container Not Visible on Host

Cause: Using a named volume instead of a bind mount, or the container is writing to a different path.

Fix: Verify with docker inspect <container> | jq '.[].Mounts' that the mount type is "bind", not "volume". The source should be your host path, not a Docker volume name.

Best Practices for Workspace Mounts

Use relative paths in compose

./:/workspace is portable across machines. Avoid hardcoding /home/alice/projects/... — it breaks on other machines.

Always set UID/GID explicitly

Never rely on the container's default UID. Use WANTED_UID and WANTED_GID environment variables. This is the #1 cause of support issues.

Prefer :Z or :z for SELinux

On RHEL/Fedora/CentOS with SELinux enforcing, append :Z to the mount: ./:/workspace:Z. Without it, the container can't read any files.

Don't mount over critical container paths

Never mount ./:/ (root). You'll shadow /bin, /etc, and /usr — the container won't start. Always mount to a dedicated path like /workspace.

FAQ — docker_mount_cwd_to_workspace

Does docker_mount_cwd_to_workspace work on macOS?

Yes, but with caveats. Docker Desktop on macOS uses a Linux VM — bind mounts go through osxfs, which is slower than native Linux mounts. For large projects (10,000+ files), expect a 2-3x slowdown on file operations. Mitigation: use :cached or :delegated mount options for better performance at the cost of consistency.

Can I mount multiple directories into /workspace?

Docker doesn't support merging multiple host directories into one container path. If you need files from two locations, mount them to separate paths (/workspace and /data) and use symlinks inside the container.

What if my project is on an external drive?

Docker Desktop: add the external drive path in Settings → Resources → File Sharing. Linux: make sure the drive is mounted with exec permissions (check /etc/fstab for noexec flag).

How does this relate to the Hermes WebUI bootstrap error?

The bootstrap Python environment error often appears alongside volume mount problems because the agent source needs to be accessible inside the container. If /workspace or the hermes-agent mount is empty, the bootstrap check fails. See the main error catalog for the full fix.
